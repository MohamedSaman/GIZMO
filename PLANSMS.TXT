SMS Management System — Implementation Plan
Overview
Build a full SMS tracking and billing system with:

Customer-facing SMS section in Settings page — customers see their total sent SMS count, topup balance, and can filter by month.
Admin SMS Integration section in Settings — shows each customer's cost & topup balance.
Secret super-admin-only page (accessible via a secret URL only you know) — set SMS per-message rate, view system-wide SMS counts, top up customer balances.
Low-balance warning SMS — when a customer's balance drops below a threshold, automatically send them an SMS alert to their phone.
Double-charge logic — when balance hits zero, SMS cost doubles automatically.
User Review Required
IMPORTANT

Secret Page URL: I'll create a secret URL like /admin/sms-master-control/{secret_token} that only you know. The token will be stored in .env. Please confirm you're okay with this approach.

IMPORTANT

SMS Logs Table: There is no existing sms_logs table. I'll create a migration for it to track every sent SMS (customer_id, phone, message, cost, sent_at). Is this okay?

WARNING

Customer "Topup" Concept: Currently customers don't have an SMS credit balance. I'll add sms_balance and sms_rate fields. Do customers topup manually (you add credits) or pay automatically?

Open Questions
IMPORTANT

Which "customer" are we talking about? The system has both customers (buyers in the store) and users (staff/admin logins). Who sees their SMS balance — the customers table people (who receive invoice SMSes) or users (staff who log in)?

NOTE

Based on context (invoice SMS sent to customers), I'll assume customers table is the right entity. Their phone receives SMSes, and they have a topup balance that gets deducted per SMS.

NOTE

Low balance threshold: I'll use a configurable threshold (default: 50 LKR). When balance drops below this, an alert SMS is sent to the customer's phone.
